# Логистическая модель запуска ABC4RD Academy

Статус: русский рабочий документ. Это не юридическая, налоговая или
инвестиционная консультация. Это операционная карта запуска.

Цель документа: разложить весь запуск Академии по разделам и подразделам, чтобы
было видно, что уже есть, что нужно сделать дальше, где проходят студенты, где
формируются карточки, как идут коммуникации, как запускается сайт, как не
торопиться с регистрацией компании и как проверить модель на первых тестах.

## 2. Главная логика запуска

### 2.1 Не регистрировать компанию первым действием

Первым действием не является регистрация компании.

Первым действием является запуск проверяемой образовательной системы:

- сайт;
- воронка студентов;
- первый курс;
- первая группа;
- первый demo;
- первый review;
- первый сертификат;
- первый отчет по результатам.

Компания нужна тогда, когда появляются:

- регулярные платежи;
- B2B contracts;
- payroll или contractor payments;
- налоговая обязанность;
- партнерские договоры;
- необходимость банковского счета;
- необходимость юридически принимать sponsorship/grants.

### 2.2 Как запускаться до компании

До регистрации компании запуск идет как public pilot:

- бесплатные материалы;
- waiting list;
- заявки на interest;
- тестовые cohort;
- test certificate;
- no paid public checkout без approval;
- no crypto acceptance без approval;
- no institutional contract без review.

Можно собирать:

- интерес;
- заявки;
- feedback;
- партнерские leads;
- source suggestions;
- volunteer/contributor requests.

Нельзя без review:

- принимать массовые платежи;
- обещать сертификаты за деньги;
- принимать crypto на личный wallet как оплату Академии;
- публиковать territory-specific paid offers;
- подписывать договоры от имени несуществующей структуры.

### 2.3 Когда включать деньги

Деньги включаются по стадиям:

| Стадия | Что можно делать | Что нельзя делать |
| --- | --- | --- |
| Stage 0: public pilot | free access, waitlist, тестовые заявки | платный checkout |
| Stage 1: manual paid pilot | один approved fiat rail, ручные invoices | автоматический массовый прием |
| Stage 2: operating entity | bank account, contracts, accounting | хаотичные личные платежи |
| Stage 3: scaled academy | cohorts, B2B, grants, sponsorships | запуск новых направлений без owner |

- фиксировать demand через заявки;
- подготовить payment rails decision;

### 3.4 Как деньги распределяются

Treasury buckets:

- operating reserve;
- teachers/contractors;
- scholarship fund;
- crypto experiment, если approved;
- emergency reserve.

### 3.5 Что нужно сделать практично

Открытые решения:

- выбрать fiat payment rail;
- решить, кто юридически принимает деньги;
- решить, crypto off/pilot/enabled;
- создать invoice/receipt template;
- создать scholarship ledger;
- создать простую таблицу revenue/events.

## 4. Сайт

### 4.1 Где делать сайт

Первый сайт должен быть простым и управляемым.

Базовое решение:

```text
GitHub repository -> static website -> GitHub Pages / static hosting
```

Почему:

- все материалы уже в GitHub;
- история изменений прозрачна;
- легко делать PR;
- можно быстро запускать;
- не нужен сложный backend в первый день;
- можно сохранить English canonical source и русские страницы рядом.

### 4.2 Что должно быть на сайте

Первый сайт должен отвечать на вопросы за 30 секунд:

- кто мы;
- почему нам можно верить;
- что есть с 2017 года;
- что такое ABC4RD Academy;
- какие языки;
- какой первый курс;
- какой первый demo;
- как попасть в cohort;
- как получить test certificate;
- как стать contributor;
- как связаться.

### 4.3 Первая структура сайта

Главные разделы:

- Home;
- About / Background;
- Start Here;
- Courses;
- Labs;
- Certificates;
- Research Bulletin;
- Languages;
- Pricing / Financial Model;
- Partners / Institutions;
- Apply / Join;
- Contact.

### 4.4 Что делать не сразу

Не начинать с тяжелого портала.

Позже:

- личный кабинет;
- student dashboard;
- автоматические payments;
- LMS;
- CRM portal;
- certificate verification automation;
- multi-language CMS.

## 7. Карточка студента

### 7.1 Где формируется карточка

Карточка студента должна формироваться не в GitHub issue, а в CRM или CRM-like
таблице.

Первый возможный уровень:

- private spreadsheet;
- Airtable-like table;
- NocoDB/Baserow;
- Twenty/EspoCRM/Corteza;
- позже полноценный CRM cockpit.

GitHub хранит public learning artifacts и issues, но не private student data.

### 7.2 Поля карточки студента

Минимальные поля:

- student ID;
- name;
- email;
- language group;
- timezone;
- intake source;
- selected offer;
- payment status;
- scholarship status;
- cohort status;
- course progress;
- demo status;
- review status;
- certificate status;
- communication channel;
- next action;
- owner.

### 7.3 Статусы студента

```text
lead
  -> applied
  -> accepted
  -> payment/scholarship pending
  -> onboarded
  -> active learner
  -> submitted evidence
  -> review
  -> approved / revise / rejected
  -> certificate issued
  -> alumni / contributor
```

### 7.4 Кто ведет карточку

Owner нужен на каждом этапе:

- intake owner;
- cohort owner;
- mentor/reviewer;
- certificate reviewer;
- finance owner;
- community owner.

CRM должна быть не просто sales database.

Она должна соединять:

- организации;
- преподавателей;
- reviewers;
- заявки;
- платежные статусы;
- cohort status;
- communication history;
- tasks;
- documents;
- certificate review.

### 8.2 Первый practical вариант

На первом этапе:

- простая private таблица;
- формы заявок;
- ручное назначение owner;
- связь с GitHub issues;
- еженедельный review.

На втором этапе:

- Twenty / EspoCRM / Corteza / NocoBase / ERPNext review;
- Chatwoot для website chat/email intake;
- n8n для автоматизации;
- Nextcloud для документов;
- Jitsi/BigBlueButton/LiveKit для занятий.

### 8.3 Что не хранить в GitHub

В GitHub нельзя хранить:

- private student data;
- платежные данные;
- private messages;
- договоры без redaction;
- tokens;
- keys;
- passport/KYC documents;
- конфиденциальные partner notes.

## 11. Тест на себе

### 11.1 Test 01: Один студент

Провести одного тестового студента через весь путь:

1. заходит на сайт;
2. понимает Академию за 30 секунд;
3. подает заявку;
4. получает карточку;
5. проходит onboarding;
6. открывает Course 01;
7. запускает demo;
8. отправляет evidence;
9. получает review;
10. получает test certificate или revise decision.

### 11.2 Test 02: Одна организация

Провести одну тестовую организацию:

1. видит site/partner brief;
2. попадает в CRM как organization;
3. получает discovery call template;
4. получает institutional starter proposal;
5. получает finance route;
6. получает course/demo/certificate workflow;
7. получает final report.

### 11.3 Что измеряем

Измеряем:

- где человек теряется;
- где непонятна ценность;
- где не хватает CTA;
- где слишком много текста;
- где нужна карточка;
- где нужна автоматизация;
- где возникает финансовый вопрос;
- где нужны юридические ограничения;
- сколько времени занимает путь.
