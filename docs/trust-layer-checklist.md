# Trust Layer Checklist

Use this checklist before treating any ABC4RD Academy repository as public
infrastructure.

## Identity

- [ ] Repository purpose is clear in the first paragraph.
- [ ] Audience is named.
- [ ] Status is visible: `draft`, `active`, `review`, or `reviewed`.
- [ ] Related repositories are linked.

## Sources

- [ ] `sources.md` exists.
- [ ] Source statuses are recorded.
- [ ] Unverified claims use `requires verification`.
- [ ] License/access status is recorded for reused materials.
- [ ] No pirated PDFs, copied standards, or unclear mirrors are included.

## Education

- [ ] At least two modules are non-placeholder.
- [ ] Each module has learning goals.
- [ ] Each module has student exercises.
- [ ] Glossary exists.
- [ ] Expected outputs are described.

## Contribution

- [ ] `CONTRIBUTING.md` exists.
- [ ] Issues are open and concrete.
- [ ] PR template exists.
- [ ] Beginner tasks exist.
- [ ] Contribution ladder is linked.

## Legal And Safety

- [ ] License is present.
- [ ] Disclaimer is present.
- [ ] No medical advice claims.
- [ ] No financial/investment advice claims.
- [ ] No safety-critical robotics deployment claims.
- [ ] No nanomaterials handling instructions.
- [ ] Contributor privacy is respected.

## Reputation

- [ ] Recent digest exists.
- [ ] Changelog or release notes exist.
- [ ] At least one issue was closed after real work.
- [ ] External contribution targets are identified.
- [ ] Repository has topics and a concise description.

## Decision

- `pass`: repository can be promoted.
- `needs work`: repository is public but should not be featured yet.
- `hold`: repository needs IP, legal, safety, or source review before promotion.
