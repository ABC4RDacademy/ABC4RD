# Outreach Pipeline

ABC4RD Academy outreach is careful, narrow, and source-backed. The goal is
accuracy feedback and useful collaboration, not promotion.

## Outreach Stages

| Stage | Public proof required | Allowed action |
|---|---|---|
| 0. Watch | Official source or repo identified | Add to map or watchlist |
| 1. Verify | Source-review issue exists | Mark status and relevance |
| 2. Use | Source appears in curriculum, digest, or reading list | Cite with attribution |
| 3. Contribute | Small docs issue or PR identified | Open respectful upstream issue/PR |
| 4. Ask | ABC4RD artifact is useful and public | Ask for accuracy correction |
| 5. Collaborate | Relationship exists | Invite guest session or review |

## What Not To Do

- Do not ask for partnership first.
- Do not imply endorsement.
- Do not mass message speakers or sponsors.
- Do not scrape private contacts.
- Do not paste private replies into public repos.
- Do not use Bitrix as a source of public claims.

## Message Types

| Type | When to use |
|---|---|
| Source correction request | After a public ABC4RD note cites an official source |
| Documentation PR note | When opening a small upstream docs PR |
| Preferred link request | When an organization has several public resource pages |
| Guest session invitation | Only after public materials are credible |
| Thank-you note | After referencing a public talk or paper |

## Bitrix Routing

Every meaningful reply should become a Bitrix task or contact note:

- source correction;
- collaboration suggestion;
- event lead;
- university lead;
- contributor offer;
- external PR review;
- legal/IP concern;
- medical/safety concern.

Do not store secrets, private tokens, or confidential documents in GitHub.
